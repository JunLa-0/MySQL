# 引言

`MySQL`数据库的核心功能就是存储数据，通常是整个业务系统中最重要的一层，如果`MySQL`数据库存在些许问题，那么对于整个系统而言都是致命的

那么`MySQL`在接受外部数据写入的时候，会发生问题吗？

- 对于`MySQL`本身而言，在写入数据的时候是不会有问题的，因为`MySQL`的数据是存储到磁盘中的，就算本地部署`MySQL`的机器宕机/断电了，它内部都有一套健全的机制确保数据不丢失

- 但是如果结合业务来看，就会有很大问题，因为涉及到了具体的业务场景，可以看一个案例：`看一个比较常见的下单案例`

  - 经典的下单业务，用户下单，然后系统进行库存扣减，扣减成功后添加订单记录，然后插入相应的物流信息，这就是一个正常的下单业务流程

  ~~~sql
  # 从库存表中扣减对应商品的库存
  update user_inventory set ......;
  
  # 向订单表中新增订单记录
  insert into user_order values(....);
  
  # 向物流表中插入相应的物流信息
  insert into user_logistics values(....);
  ~~~

  **那么结合了实际业务场景来看会出现什么问题呢？**

  因为在实际项目开发中，这三条`SQL`是通过客户端(假设我们这里是`Java`线程)来维护的，并且通过客户端来一条条发送过来的，假设执行到新增订单记录的时候，`Java`程序抛出了异常，那么`Java`线程就中断了，那么线程就不会向数据库发生新增订单记录和物流信息这两条`SQL`，那么数据库就没法执行这两条`SQL`，但是用户已经下单了，钱也付了，但是没有订单信息和物流信息，没有保证事务一致性，那么这个程序员就该死了

> 上面的问题就是一个在`MySQL`结合事务去研究写入数据会发生的问题，就是一个典型的问题

# 事务的ACID原则

**什么是事务**
事务通常是由一条或一组`SQL`(多条`SQL`的意思)组成的，组成一个事务的`SQL`一般都是一个业务操作，就是一个业务逻辑的实现(一个业务功能的实现)，例如前面说的下单：`用户下单，扣减对应商品库存，生成订单信息，发生物流，生成对应物流信息`，这一组`SQL`就是一个业务操作，就组成了一个事务

数据库的事务一般都要满足`ACID`原则，`ACID`原则是关系型数据库实现事务机制时必须要遵守的原则

`ACID`的四条原则：

- `A/Atomicity`：原子性——在一个事务中，所有的操作要么全部执行成功，要么全部执行失败
- `C/Consistency`：一致性——事务发生前后数据库中的数据都处于一致性的状态
- `I/Isolation`：隔离性/独立性——事务之间是隔离的，互不影响的
- `D/Durability`：持久性——一个事务一旦被提交了，那么此时的操作就会保持永久性

下面就来聊聊这四条原则

-----

## Atomicity原子性

`MySQL`中的原子性的含义和`Java并发编程中JMM内存模型`中学习到的原子性大致相同，指组成一个事务的一条或一组`SQL`要么全部执行成功，要么全部执行失败，就是把一个事务就当成是一个操作来看待

> 例如：
> 事务`A`由`1、2、3`这三条`SQL`组成，那么就是这个事务有三条`SQL`，组成了一个业务操作，那么要满足一致性，这三条`SQL`必须全部执行成功，如果三条`SQL`中的其中一条执行失败了，就会导致三条`SQL`全部执行失败，即导致事务`A`的所有操作都失败，因为要满足事务原子性，全部执行成功或全部执行失败

-----

## Consistency一致性

一致性指的就是事务发生前后，`MySQL`中原本的数据变化都是一致的，就是数据库中的数据只能从一个一致性状态变为另一个一致性状态，简单点说就是一个事务中的所有操作，要么一起改变数据库中的数据，要么都不改变

还是以订单业务为例子：

> 假设有一个事务`A`，这个事务有三条`SQL`组成，分别是库存扣减、增加订单详情记录、插入物流信息

那么一致性指的就是：在这个事务执行钱数据库中的数据处于一致性的状态，这组`SQL`操作执行之后进行事务提交之后，数据库中的数据依然处于一致性的状态，对于这个事务而言也就是在执行操作前数据库的总库存是`10000`个，那么执行完`SQL`操作之后，总库存剩余`8888`个，那么订单数量必须是`1112`条订单数据

> 这就是事务一致性，执行`SQL`操作前后`MySQL`数据库中的数据都是保持一致性的，不可能出现库存扣减了，没有订单信息，这样子就导致数据库整体数据出现不一致，不符合事务一致性

但是如果真的出现了不符合一致性的情况怎么办呢，即库存扣减了，但是订单没有增加，没有新增订单信息的情况，那么`MySQL`就会利用事务回滚机制，将扣减的库存数量增加回去，先确保事务一致性

> 但是如果在执行事务的过程中，库存成功扣减之后，没法新增订单信息，然后部署`MySQL`的服务器也断电/宕机了，就没法用事务回滚机制去恢复数据保证数据一致了
>
> 这个问题并不需要担心，因为即使断电/宕机了，`MySQL`会通过分析日志的方式恢复数据，确保一致性，这点在后面会说

----

## Isolation隔离性/独立性

隔离性指的就是每一个事务之间是隔离的，独立的，就相当于每一个事务都被封装到了一个互相隔离的箱子中，相互之间互不影响，互不依赖，举个例子：还是下单的案例

> 假设数据库中库存表中库存数量剩余`8888`个，然后此时有`A、B`两个并发事务，这两个事务的业务逻辑都一样，都是库存扣减、增加订单详细记录、插入物流信息这三条`SQL`操作组成

如果这两个事务同时执行，同一时刻执行扣减库存的`SQL`，`因为它们同时执行相同的操作，所以这两个事务称为并发事务`，会导致数据库中库存表中库存剩余数量扣减两次吗，很显然不会

这两个事务之间是没有任何关系的，是隔离的，在隔离性中保证了两个并发事务的顺序执行，一个未完成的事务不会影响另外一个未完成的事务

> 隔离性是基于`MySQL`的锁机制和`MVCC`机制实现的，在后续`MySQL事务与锁原理篇`讲解

------

## Durability持久性

持久性指的就是：一个事务一旦被提交了，那么此时的操作就会保持永久性，所更改的数据都会被写入到磁盘中做持久化处理，就算`MySQL`宕机了也不会影响数据的改变，因为宕机后可以通过日志恢复数据

也就是说只要你这个事务提交了，不管发生什么情况，都要根据你提交的事务对数据的修改，更改到数据库中，写入到磁盘中进行持久化处理

-----

# MySQL的事务机制综述

在`MySQL`默认情况下，一条`SQL`就会被视为一个单独的事务，而且不需要我们手动提交，因为默认是开启事务自动提交机制的，如果想将多条`SQL`组成一个事务进行执行，那么就要手动去显式地通过一些事务指令来实现

----

## 手动管理事务

在`MySQL`中提供了一系列事务相关的指令，如下：

- `start transaction | begin | begin work`：开启一个事务
- `commit`：提交一个事务
- `rollback`：回滚一个事务

当需要使用事务时，可以先通过`start transaction`命令开启一个事务，如下：

~~~sql
# 开启一个事务
start transaction;

# 执行事务操作
-- 第一条sql
-- 第二条sql
-- 第三条sql
# 这个事务执行完毕

# 提交或回滚事务
# 没问题就提交，不满足一致性就回滚
commit || rollback;
~~~

事务是基于数据库连接而言的，不是基于表的，一个事务中的`SQL`可以操作不同的表，即一个事务可以由操作不同表的多条`SQL`组成，前面也说过，一条`SQL`或一组`SQL`组成事务，从上面的下单业务就可以看出来，在这个事务中的这组`SQL`中，即操作了库存表，也操作了订单表，还操作了物流信息表

我们可以再举一个例子看看：

## 事务回滚点

假设现在有一个事务，这个事务由很多条`SQL`组成，我们想让这些`SQL`一部分执行成功后，就算后续的`SQL`执行失败也照样提交事务，根据前面所说的事务的原子性，似乎不能实现这个需求，但实际上我们可以利用事务的回滚点机制来实现

> 在某些`SQL`执行成功后，这个事务中后续的`SQL`可能执行成功，可能执行失败，但是不管成功还是失败，我都想让前面已经执行成功的操作生效，此时就可以在当前执行成功的位置设置一个回滚点，当后续操作执行失败时，就可以回滚到这个设置回滚点的位置，而不是回滚真个事务，导致前面执行成功的操作也失效，这个机制就称为事务回滚点

在`MySQL`中提供了两个关于事务回滚点的命令：

- `savepoint point_name`：添加一个事务回滚点
- `rollback to point_name`：回滚到指定的事务回滚点

通过前面讲述的案例来演示一下：

~~~SQL
# 先查询一下表数据
select * from user_table;

# 手动开启事务
start transaction;

# 在手动提交事务前后续的sql都是这个事务中的sql
update user_table set user_name = "黑熊" where user_id = 4; # 修改

# 上面的sql执行成功后，可以在执行成功后的位置加一个事务回滚点以供后续回滚选择
savepoint update_name;

# 进行删除
delete from user_table where user_id = 1;

# 我现在只想update执行成功后就提交事务，不管后续的delete是否成功我都要提交事务
# 那么就要使用回滚点进行回滚
rollback to update_name;

# 然后再查询一次数据，会发现只有update命令生效了
select * from user_table;

# 提交事务
commit;
~~~

> 注意：回滚到事务点后不代表事务结束了，只是事务内发生了一次回滚，如果要结束当前这个事务还需要进行`commit/rollback`命令进行处理

> 理论上，通过事务回滚点机制，可以很好地实现失败重试，就是事务中对每个`SQL`都添加一个回滚点，当执行到某一条`SQL`时失败了，就回滚到上一条`SQL`的事务点，然后再次执行失败的`SQL`,反复执行到所有`SQL`成功为止，最后提交整个事务
>
> 这只是理论，实际工作不要这么干

## MySQL事务的隔离机制

事务是基于数据库连接的，而不是基于表的，一个事务的连接不会影响其他连接，这是基于事务隔离级别实现的，那么接下来我们就看看事务隔离级别，在`MySQL`中事务隔离级别有四个级别：

- `Read uncommitted/RU`：读未提交
- `Read committed/RC`：读已提交
- `Repeatable read/RR`：可重复读
- `Serializable`：序列化/串行化

上述四个隔离级别，越靠后的并发控制度越高，也就是在多线程并发操作的情况下，出现问题的概率越小，但对应的性能也差，`MySQL`的事务隔离级别默认是第三级别：`Repeatable read`可重复读，想要理解这四个隔离级别，得先明白几个因为并发操作造成的问题

-----

### 脏读、幻读、不可重复读问题

#### 数据库的脏读问题

脏读就是一个事务读到了其他事务未提交的数据，`正常情况下，一个事务和其他事务是互不影响的，这个事务的数据修改，在没有提交前，其他事务是查询不到更改的，这个在前面已经说过了`

> ![脏读](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/721f479d93ec45ee8d76757ebaec8c44~tplv-k3u1fbpfcp-jj-mark:3024:0:0:0:q75.awebp)

数据库连接1`事务A`正在执行一个下单的业务，扣减库存和增加订单两条`SQL`已经执行成功了，但是此时数据库连接2`事务2`过来读取了一下库存剩余数量，就将`事务A`已经扣减库存后的数据读取了，但是由于`事务A`在添加物流信息的时候出现了异常，所以进行了全部回滚，那就是说库存数量又变回了原来的样子，但是`事务B`读取到的确实库存扣减后的数据，那么这两个事务的数据是不一样的，这个过程就是数据库脏读问题，会导致整个业务系统出现问题，导致数据错乱

----

#### 数据库的不可重复读为题

不可重复读：指的是一个事务中多次读取同一个数据，先后读取到的数据不一致：

> ![不可重复读问题](https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/2bcea9831f4a4853b2750fce1bf4ac12~tplv-k3u1fbpfcp-jj-mark:3024:0:0:0:q75.awebp)

这个图相对于上面的图进行了一下修改，`事务A`执行下单业务时，因为在添加物流信息的时候出现了异常所以进行了整个事务的回滚操作，事务回滚完了之后`事务A`就结束了，大那是`事务B`还没有结束，所以`事务B`在`事务A`回滚前读取了一次数据，然后在`事务A`回滚后又读取了一次数据，`事务B`读取到的数据分别是库存扣减前和库存扣减后的数据

在同一个事务中读取同一数据但是结果却不一致，也就说明了该数据存在不可重复读的问题

> 不可重复读：就是读取同一个数据但是结果不一样，都是读取库存数量，但是两次读取的结果不一样
>
> 可重复读：读取同一个数据，无论读取多少次结果都一样，都是读取库存数量，不管读取多少次读取出来的结果都一样

> 例如：一张纸巾擦了污渍之后返回原位置，但是你想用它去擦嘴已经无法使用了，就是不可重复使用

> 一个铁锤敲完木板还可以用来敲钉子，可重复使用

-----

#### 数据库的幻读

幻读指的是：

> ![幻读问题](https://p6-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/92d0c9cf2fbc4441ab6d9c2c579c6751~tplv-k3u1fbpfcp-jj-mark:3024:0:0:0:q75.awebp)

我们举一个电商系统的例子，一般用户购买商品后付的钱会先冻结在平台上，然后由平台在固定的时间内进行结算用户款，例如七天一结算、半月一结算等方式，在结算业务中通常都会涉及到核销处理，也就是将所有为已签收的状态的订单改为已核销状态

假设`事务A`正在执行半月核算这个工作，首先会读取订单表中所有的状态为已签收的订单，然后将这些订单更改为已核销状态，然后将用户款打给商家

但是恰巧，某个用户的订单正好到了自动确认收货的时间，因此在`事务A`刚改完表中订单的状态时，`事务B`又向表中插入一条已签收状态的订单并且提交了，当`事务A`完成打款后，再次去查询订单表，会发现表中还有一条已签收状态的订单数据未结算，这就好像产生了幻觉，这就是幻读

> 再举个例子：
>
> 假设某个平台需要进行升级改进，用户表中的性别字段要从男/女形式保存数据改成0，1形式保存数据
>
> 因此`事务A`开始更改表中所有的数据的性别字段，当负责执行`事务A`的线程正在更改最后一条数据时，此时`事务B`来了，正好向用户表中插入了一条性别=男的数据，并且提交了，然后`事务A`修改完最后一条数据后，再次去查询用户表的时候，发现依旧还有一条性别=男的数据，似乎出现了幻觉一样，这就是幻读

幻读产生的原因就是：另外一个事务在第一个事务要处理的目标数据范围内新增了数据，然后优于第一个事务提交数据造成的问题

----

#### 数据库脏写问题

除了上面的三个问题，其实还有一个脏写的问题，也就是多个事务一起操作同一条数据，例如两个事务同时向表中添加一天`ID=88`的数据，此时就会造成数据覆盖，或主键冲突的问题，这个问题也会被称为更新丢失问题

----

### 事务的四大隔离级别

上面讲了脏读、幻读、不可重复读这三个问题，这些问题是如何解决的呢？

其实四个事务隔离级别就是用来解决这三个问题的，那么我们来看看这四个事务隔离级别分别解决了什么问题：

- 读未提交：处于该事务隔离级别的数据库，脏读、幻读、不可重复读都有可能发生
- 读已提交：处于该事务隔离级别的数据库，解决了脏读，幻读、不可重复读问题依然存在
- 可重复读：处于该事务隔离级别的数据库，解决了脏读、不可重复读问题，幻读问题依然存在
- 序列化/串行化：处于该事务隔离级别的数据库，解决了脏读、幻读、不可重复读问题，这些问题都不会出现了

`MySQL`默认是处于第三级别的，即可重复读的级别，可以通过如下命令查看当前数据库的隔离级别

~~~sql
# 查询方式
select @@tx_isolation;

# 查询方式
show variables like '%tx_isolation%';
+---------------+-----------------+
| Variable_name | Value           |
+---------------+-----------------+
| tx_isolation  | REPEATABLE-READ |
+---------------+-----------------+
~~~

数据库不同的事务隔离级别是基于不同类型、不同粒度的锁实现的，因此想要真正理解事务隔离级别，还需要明白`MySQL`的锁机制，在后续的文章中会详细讲解`MySQL`的锁机制，事务和锁机制二者之间本身就是相辅相成的，锁就是为了解决并发事务的一些问题而存在的

-----

#### 读未提交级别

这种隔离级别是基于`写互斥锁`实现的，当一个事务开始写某一个数据的时候，另外一个事务也来对这个数据进行操作，就是这两个事务操作同一个数据，那么此时为了防止出现问题就需要先获取锁资源，只有获取到锁的事务，才允许对数据进行写操作，而且获取到锁的事务具备排他性/互斥性，也就是其他线程无法再操作这个数据，只允许获取到锁的事务对这个数据进行操作

**注意：
在写同一个数据的时候会互斥，但是在读操作是不会互斥的，也就是当一个事务在写某个数据的时候，不管有没有提交事务，其他事务来读这个数据的时候，是可以读到未提交的数据的，所以就会出现脏读、不可重复读、幻读这一系列问题**

但是在这个事务隔离级别中，添加了`写互斥锁`，即不会出现多个事务同时对同一个数据进行操作的情况，所以解决了脏写的问题

----

#### 读已提交级别

在这个事务隔离级别中进行写操作的时候一样会使用`写互斥锁`，即两个事务操作同一个数据的时候会出现排他性，而对于读操作采用了`MVCC`多版本并发控制的技术处理，也就是有事务中的`SQL`需要读取当前事务正在操作的数据时，`MVCC`机制不会让另外的事务来读取这个正在被操作的数据，而是读取上一次提交的数据，也就是读取原本的老数据，是读取不到此时正在修改的数据的

> 也就是说，在这个事务隔离级别中，对于同一条数据而言，写操作有排他性，读操作只能读取到已提交事务的数据，不会读取到正在操作且未提交的事务数据，那么就解决了脏读的问题

> 举个例子吧：
>
> 有两个事务，事务`A`和事务`B`
>
> 事务`A`负责更新`id=1`的数据，事务`B`读取`id=1`的数据，此时`A`正在更新数据但是还没有提交，事务`B`开始去读取数据，此时`MVCC`机制会基于表数据的快照创建一个`ReadView`，然后读取原本表中上一次提交的老数据，然后事务`A`提交之后，事务`B`再次读取数据，此时`MVCC`机制又会创建一个新的`ReadView`，然后读取到最新的已提交的数据，所以事务`B`两次读取到的数据是不一致的，所以会出现不可重复读的问题，但是解决了脏读的问题

**MVCC机制和锁机制后续单独讲解**

----

#### 可重复读级别

主要是解决上一个级别中遗留的不可重复读问题，但`MySQL`依旧是使用`MVCC`机制来解决这个问题，这个`MVCC`机制和上面说的有区别，在读已提交级别中，一个事务中每次查询数据时都会创建一个新的`ReadView`，然后读取最新提交的事务数据，所以就造成了不可重复读的问题

但是在可重复读级别中，不会每次查询数据时都新建一个`ReadView`，而是在一个事务中，只有第一次执行查询会创建`ReadView`，在这个事务的生命周期内，所有的查询都会从这个`ReadView`中读取数据，从而确保一个事务中多次读取相同数据是一致的，就解决了不可重复读的问题

------

#### 序列化/串行化级别

这个隔离级别是最高的级别，处于该级别的`MySQL`一定不会出现任何问题

序列化的意思就是将所有的事务按序排队后进行串行化处理，也就是操作同一张表的事务只能一个一个执行，事务在执行前需要先获取表级别的锁资源，拿到锁资源的事务才能执行，其余事务就陷入阻塞，等待当前拿到锁资源的事务释放锁资源

但这个事务隔离级别的会导致数据库的性能直线下降，因为这个事务隔离级别相对于一张表只能运行单线程执行，基本很少用这个事务隔离级别

在前面分析：出现这些问题的原因就是多事务/多线程并发执行导致的，那我直接吧多线程变成单线程不就好了

### 事务隔离机制的命令

简单看一下事务隔离级别相关的命令：

**如果想要设置的隔离级别在全局生效，记得加上`global`关键字，如果不加这个关键字，设置的隔离级别只在当前会话`当前session`中生效，即针对当前数据库连接有效`只针对当前连接生效`

~~~sql
# 方式1：查询当前数据库的隔离级别
SELECT @@tx_isolation;

# 方式2：查询当前数据库的隔离级别
show variables like '%tx_isolation%';

-- 设置隔离级别为RU级别（当前连接生效）
set transaction isolation level read uncommitted;

-- 设置隔离级别为RC级别（全局生效）
set global transaction isolation level read committed;

-- 设置隔离级别为RR级别（当前连接生效）
-- 这里和上述的那条命令作用相同，是第二种设置的方式
set tx_isolation = 'repeatable-read';

-- 设置隔离级别为最高的serializable级别（全局生效）
set global.tx_isolation = 'serializable'; 

~~~

# MySQL的事务实现原理

结论：**MySQL的事务机制是基于日志实现的**

----

## 正常SQL的事务机制

**默认情况下，MySQL开启事务的自动提交，并且将一条SQL当作一个事务**

那么`MySQL`在什么情况下会将事务自动提交呢？什么情况下会自动回滚呢？

我们讲过：任何一条写`SQL`的执行都会记录三个日志：`undo-log、redo-log、bin-log`

- `undo-log`：主要记录`SQL`的撤销日志，比如这是一条`insert`语句，那么就会记录一条`delete`日志
- `redo-log`：记录当前`SQL`归属事务的状态，以及记录修改内容和修改页的位置
- `bin-log`：记录每条`SQL`操作日志，主要是用于数据的主从赋值和数据恢复/备份

在写`SQL`执行记录的三个日志中，`bin-log`暂且不需要关心，这个跟事务机制没关系，重点关注`undo-log、redo-log`这两个日志，其中最重要的是`redo-log`这个日志

> `redo-log`是一种`WAL(write-ahead logging)`预写式日志，在数据发生更改前会先记录日志，也就是在`SQL`执行前会先记录一条`prepare`状态的日志，然后再执行数据的写操作

但要注意：`MySQL`是基于磁盘的，但磁盘的写入速度相较内存而言会较慢，因此`MySQL-InnoDB`引擎中不会直接将数据写入到磁盘文件中，而是会先写到`BufferPool`缓冲区中，当`SQL`被成功写入到缓冲区后，紧接着会将`redo-log`日志中相应的记录改为`commit`状态，然后再由`MySQL`刷盘机制去做具体的落盘操作。

> 因为默认情况下，一条`SQL`会被当成一个事务，数据写入到缓冲区后，就代表执行成功，因此会自动修改日志记录为`commit`状态，后续则会由`MySQL`的后台线程执行刷盘动作。

举个伪逻辑的例子，例如下述这条插入`SQL`的执行过程大致如下：

```sql
sql 体验AI代码助手 代码解读复制代码-- 先记录一条状态为 prepare 的日志

-- 然后执行SQL，在缓冲区中更改对应的数据
INSERT INTO `zz_users` VALUES(5,"黑竹","男","9999","2022-09-24 23:48:29");
-- 写入缓冲区成功后，将日志记录改为 commit状态
-- 返回 [Affected rows: 1]，MySQL后台线程执行刷盘动作
```

一条`SQL`语句组成的事务，其执行过程是不是很容易理解~，接着来看看手动开启事务的实现

## 多条SQL的事务机制

先把上面的案例搬下来：

~~~sql
-- 开启事务
start transaction;

-- 修改 ID=4 的姓名为：黑熊（原本user_name = 1111）
update `zz_users` set `user_name` = "黑熊" where `user_id` = 4;

-- 删除 ID=1 的行数据
delete from `zz_users` where `user_id` = 1;

-- 提交事务
COMMIT;
~~~

1. 当`MySQL`执行时，碰到`start transaction`命令时，会将后续所有的写操作全部关闭自动提交机制，也就是后续的所有写操作，不管成功还是失败都不会将日志记录修改成`commit`状态，因为手动开启事务，设置成了手动提交事务/手动回滚事务

2. 现在`redo-log`中为第一条`SQL`语句记录一条`prepare`状态的日志，然后再生成对应的撤销日志并记录到`undo-log`中，然后执行`SQL`，将要写入的数据先更新到缓冲区

3. 再对第二条`SQL`语句做相同处理，如果有更多条`SQL`则逐条依次做出相同的处理，这里简单说一下撤销日志长啥样，如下：

   ~~~sql
   -- 第一条修改SQL的撤销日志（将修改的姓名字段从 黑熊 改回 1111）
   update `zz_users` set `user_name` = "1111" where `user_id` = 4;
   
   -- 第二条删除SQL的撤销日志（将删除的行数据再次插入）
   INSERT INTO `zz_users` VALUES(1,"熊猫","女","6666","2022-08-14 15:22:01");
   ~~~

   

4. 直到碰到了`rollback、commit`命令时，再对前面的所有写`SQL`做相应的处理

   > 如果是`commit`命令，则先将当前事务中所有的`SQL`的`redo-log`日志改为`commit`状态，然后由`MySQL`后天线程做刷盘处理，将缓冲区中的数据落入磁盘进行存储
   >
   > 
   >
   > 如果是`rollback`命令，则在`undo-log`日志中找到对应的撤销`SQL`执行，将缓冲区内更新的数据全部还原回去，由于缓冲区的数据被还原了，依次后台线程在刷盘时，依旧不会改变磁盘文件中存储的数据

## 事务的恢复机制

现在再来思考一个问题，有没有这么一种可能呢？也就是当`SQL`执行时，数据还没被刷写到磁盘中，结果数据库宕机了，那数据是不是就丢了啊？毕竟本地磁盘中的数据，在`MySQL`重启后依旧存在，但缓冲区中还未被刷到磁盘的数据呢？因为缓冲区位于内存中，所以里面的数据重启是不会存在的撒？

> 对于这个问题呢实际上并不需要担心，因为前面聊到过`redo-log`是一种预写式日志，会先记录日志再去更新缓冲区中的数据，所以就算缓冲区的数据未被刷写到磁盘，在`MySQL`重启时，依旧可以通过`redo-log`日志重新恢复未落盘的数据，从而确保数据的持久化特性。

当然，有人或许又会问：那如果在记录`redo-log`日志时，`MySQL`芭比Q了咋整？如果遇到了这个问题呢，首先得恭喜你，你的运气属于很棒，能碰到这个问题的几率足够你买彩票中五百万了~

> 玩笑归玩笑，现在回归话题本身，这个问题总不能让它存在是不？毕竟有这个问题对于系统而言也是个隐患啊，但仔细一思考，其实这个问题不必多虑，为啥？推导一下。

首先看看前面的那种情况：**数据被更新到缓冲区但没刷盘，然后`MySQL`宕机了，`MySQL`会通过日志恢复数据**。这里要注意的是：数据被更新到缓冲区代表着`SQL`执行成功了，此时客户端会收到`MySQL`返回的写入成功提示，只是没有落盘而言，所以`MySQL`重启后只需要再次落盘即可。

> 但如果在记录日志的时候`MySQL`宕机了，这代表着`SQL`都没执行成功，`SQL`没执行成功的话，`MySQL`也不会向客户端返回任何信息，因为`MySQL`一直没返回执行结果，因此会导致客户端连接超时，而一般客户端都会有超时补偿机制的，比如会超时后重试，如果`MySQL`做了热备/灾备，这个重试的时间足够`MySQL`重启完成了，因此用户的操作依旧不会丢失（对于超时补偿机制，在各大数据库连接池中是有实现的）。

但如若又有小伙伴纠结：我`MySQL`也没做热备/灾备这类的方案呐，此时咋整呢？

> 如果是这样的情况，那就只能自认倒霉了，毕竟`MySQL`挂了一直不重启，不仅仅当前的`SQL`会丢失，后续平台上所有的用户操作都会无响应，这属于系统崩溃级别的灾难了，因此只能靠完善系统架构来解决。
