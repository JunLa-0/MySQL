# 引言

锁这个词汇再编程中出现的次数非常频繁，几乎所有的主流编程语言都会具备完善的锁机制，在数据库中那必然也具备完善的锁机制。这里牵涉到一个`高并发`关键词，由于现在的计算机几乎都是多线程机器，所以单线程编写的应用无法把机器的性能榨干，即无法把机器的性能发挥到最大，所以为了榨干机器的性能，就出现了多线程技术

多线程技术就是为了榨干机器的性能`榨干CPU资源`和提升程序的并发支持性

但是多线程技术也带来了一些问题：**线程安全问题**

为了解决多线程并发执行造成的这个线程安全问题，就出现了锁机制，通过加锁执行的方式来解决这类问题

-----

# MySQL锁的由来和分类

`MySQL`客户端向`MySQL`发送的一条条`SQL`语句，实际上都可以理解成一个个单独的事务`在MySQL事务篇我们都讲过，自动提交的话就是把一条SQL当成一个事务`，事务是基于数据库连接的，而且在每一个数据库连接中`MySQL`都会派发一条工作线程来维护这个连接，即可以理解成一个事务的执行，本质上就是一条工作现成的执行，这种情况就被称为并发事务，所谓的并发事务就是指多条线程并发执行

> 在`事务篇`中讲过，多线程并发执行会导致的问题，就是脏写、脏读、不可重复读、幻读，这些问题是通过调整事务隔离级别来解决的，那为什么通过调整事务隔离级别就可以解决这些问题呢？
>
> 因为不同的事务隔离级别中，工作线程执行`SQL`语句时，用的锁粒度、类型不同

数据库锁机制就是为了解决并发事务带来的问题，主要是确保数据库中多余工作线程并发执行的数据安全性

> 注意：
> 并发事务肯定是基于同一个数据而言的，例如`事务A`在操作`数据Y`，`事务B`在操作`数据X`，那么这两个事务没有操作同一个数据，它们不是并发事务，没有共享资源自然就不会造成并发的问题
>
> 多个事务共同操作一张表、多个事务共同操作同一行数据这些情景才是并发事务
>
> 其实就是多个事务共享资源就是并发事务
>
> **不要连并发事务是什么就来学习锁机制**

---

## MySQL锁机制的分类

`MySQL`的锁机制与索引机制比较类似，都是由存储引擎负责实现的，那么就意味着不同的存储引擎支持的锁机制不一样，其实是指不同的引擎实现的锁粒度不一样。但是除了根据锁粒度来进行划分，锁还可以通过其他维度来进行划分，简单来看一下`MySQL`的锁体系：

- 以锁粒度的维度划分：
  - 表锁：
    - 全局锁：加上全局锁之后，整个数据库只允许读，不允许做任何写操作，全局锁就是给整个数据库进行了加锁操作
    - 元数据锁/`MDL`锁：基于表的元数据加锁，加锁后整张表不允许其他事务操作
    - 意向锁：这个是`InnoDB`中为了支持多粒度的锁，为了兼容行锁、表锁而设计的
    - 自增锁/`AUTO-INC`锁：这个是为了提升自增`ID`的并发插入性能而设计的
  - 页面锁
  - 行锁：
    - 记录锁/`Record`锁：也就是行锁，一条记录和一条数据是一个意思，所以行锁就是记录锁
    - 间隙锁/`Gap`锁：`InnoDB`中解决幻读的一种锁机制
    - 临建锁/`Next-Key`锁：间隙锁的升级版，同时具备间隙锁和记录锁的功能
- 以互斥性的维度进行划分：
  - 共享锁/`S`锁：不同事务之间不会相互排斥、可以同时获取的锁
  - 排他锁/`X`锁：不同事务之间会相互排斥，同时只能允许一个事务获取的锁
  - 共享排他锁/`SX`锁：`MySQL5.7`版本中新引入的锁，主要是解决`SMO`带来的问题
- 以操作类型的维度划分：
  - 读锁：查询数据时使用的锁
  - 写锁：执行插入、删除、修改、`DDL`语句时使用的锁
- 以加锁方式的维度划分：
  - 显式锁：编写`SQL`语句时，手动指定加锁的粒度
  - 隐式锁：执行`SQL`语句时，根据隔离级别自动为`SQL`操作加锁
- 以思想维度划分：
  - 乐观锁：每次执行前认为自己会成功，因此先尝试执行，失败时再获取锁
  - 悲观锁：每次执行前都认为自己无法成功，因此会先获取锁，然后再执行

**其实，本质上就共享锁、排他锁两种类型，只是加的方式不同，加的地方不同，所以就演变出了这么多锁的称呼**

----

# 共享锁和排他锁

**注意**
有些地方会把共享锁称为读锁，排他锁称为写锁，这个说法一点都不正确，完全就是错误的，读操作也可以是排他锁，即再读操作发生时，也不允许其他线程来操作，例如：`MySQL`中一条线程在读数据的时候加了一把读锁，那么此时另外的一条线程来对这个相同的数据进行写操作时，这条线程就会陷入阻塞，因为`MySQL`中一条线程在读时是不允许其他线程改的，那么此时读锁就相对于排他锁。

---

## 共享锁

共享锁就是不同事务之间不会排斥，`这个不会排斥是指读数据不操作不会排斥`，可以同时获取读锁并执行，**仅仅是指不会排斥其他事务来读取数据，其他事务想要来写数据时，是会出现排斥性的**，例如：

> `事务A`对`ID=88`的数据加了一个共享锁，此时`事务B、事务C`来读取`ID=88`这条数据，此时`事务B、事务C`是可以获取到共享锁来执行读操作的，但此时又来了一个`事务D`，这个事务想要对`ID=88`这条数据执行修改操作，此时共享锁会出现排斥行为，不允许这个事务获取锁来执行修改操作

**所以共享锁就是对读操作不排斥、对写操作有排斥性**

在`MySQL`中我们可以在`SQL`语句后加上相关的关键字来使用共享锁，语法如下：

~~~sql
# MySQL8.0以前的写法
select .... lock in share mode;

# MySQL8.0后的写法
select .... for share;
~~~

这种通过在`SQL`后面添加关键字的加锁形式，被称为显式锁`其实这就印证了真的本质上只有共享锁和排他锁两种类型，只是因为加锁的方式不同和加在不同的地方从而演变出了不同的称呼`

而实际上为数据库设置了不同的事务隔离级别后，`MySQL`也会对`SQL`自动加锁，这种形式则被称之为隐式锁

 我们来做一个关于共享锁的实验，打开两个`cmd`窗口，都与`MySQL`建立连接：

~~~sql
# 窗口1
# 手动开启事务
begin;

# 获取共享锁并查询id=1的数据
# 显式加锁，所以这个共享锁也称为显式锁，本质上是一样的，叫共享锁就行
select * from user_table where user_id = 1 lock in share mode;
~~~

~~~sql
# 窗口2
# 手动开启事务
begin;

# 获取共享锁并查询id=1的数据
select * from where user_table = 1 lock in share mode;
~~~

两个事务都是读操作，所以可以正常获取共享锁来进正常执行操作，都可以查询到相同的数据，自己建个表插入数据试一试就可以了

然后在窗口2，即在另外一个事务中，尝试修改`id=1`的这一行数据：

~~~sql
update user_table set user_name = "JunLa" where user_id = 1;
~~~

是修改不了的，当另一个事务尝试对具备共享锁的数据进行读操作时，会被共享锁排斥，所以共享锁也具备排斥性，排斥的是写操作，但是对读操作不排斥，被排斥的线程会陷入阻塞，只有当持有共享锁的事务操作结束之后这条阻塞的线程才能继续执行下去

就是当持有共享锁的线程提交事务之后，这个阻塞的线程才能继续往下执行，在`cmd`中会展示阻塞的时间是多久

----

## 排他锁

排他锁也叫独占锁，当一个线程获取到排他锁之后，会排斥其他线程，如果其他线程想要来对共享数据或对同一数据进行操作时，就必须等获取到排他锁的线程释放锁之后，并且通过线程间的锁竞争获取到锁资源才能对这个数据进行操作

而且排他锁不仅仅是针对于写操作的，对于一个读操作，我们也可以手动指定为其获取排他锁，当一个事务在读数据时，获取了排他锁，那么其他事务来读、写这条数据的时候都会被排斥，都是无法正常执行的

> 例如：`事务A`来对`id=88`这条数据加一个排他锁，此时`事务B`来加排他锁读取这条数据，`事务C`来写这条数据，都会被`事务A`排斥

在`MySQL`中可以通过如下方式获取排他锁：

~~~sql
select ... for update;
~~~

当两个事务同时去获取排他锁时，尝试读取一条相同的数据时，只有获取到了排他锁的数据才能正常操作，但是没有获取到排他锁的事务会陷入阻塞，直到获取到排他锁的这个事务执行结束了，释放锁了之后这条被阻塞的线程才能继续往下执行

**但是下面说的这种情况即使没有获取到排他锁也不会被阻塞：**
就是当一个事务获取到排他锁在读数据的时候，另外的这个事务没有获取到排他锁，这个事务就以普通的方式来读取数据，这种方式是可以立刻执行的，是不会阻塞的，那么为什么可以直接读取到数据呢?**这和一项并发控制`MVCC`机制有关**

-----

## MySQL锁机制的释放

在`MySQL`中释放锁的动作都是隐式的，因为手动释放很容易因为操作不当造成死锁的问题发生，因此多以锁的释放工作是`MySQL`自己来干的，就类似于`JVM`中的`GC`机制一样，把内存释放工作留给`JVM`自己完成

对于锁的释放时机，在不同的隔离级别中是不相同的，例如在可重复读级别中是在事务结束后才会释放，在读未提交级别中，是`SQL`执行完成后立马释放锁

----

那么接下来我们看看`MySQL`中不同粒度的锁，即表锁、行锁、页锁等

# MySQL表锁

表锁是最常见的一种锁，因为使用起来比较简单，应用范围也比较广，几乎所有的存储引擎都支持这个粒度的锁，比如常用的`MyISAM、InnoDB、Memory`等各大存储引擎

虽然不同存储引擎的表锁在实现上也是有一点区别的，但是表锁的意思就是以表作为锁的基础，将锁加在表上，一张表只能存在一个同一类型的表锁

不同的存储引擎的表锁在使用方式上也是不同的，比如`InnoDB`是一个支持多粒度锁的存储引擎，它的锁机制是基于聚簇索引实现的，当`SQL`执行时，如果能在聚簇索引命中数据，则加的是行锁，如果无法命中聚簇索引的数据则加的是表锁，比如:

~~~SQl
select * from user_table for update;
~~~

这条`SQL`无法命中聚簇索引，此时自然加的就是表级别的排他锁，但这个表级锁，并不是真正意义上的表锁，是一个伪装锁，但作用是相同的，锁了整张表

而`MyISAM`存储引擎，它不支持聚簇索引，所以无法以`InnoDB`的这种形式去对表上锁，因此如果要在`MyISAM`存储引擎中使用表锁，就需要使用额外的语法，如下：

~~~sql
# MyISAM存储引擎中获取读锁，具备读-读可共享性，也就是多个事务进行读操作是可以共享的
lock tables table_name read;

# MyISAM存储引擎获取写锁，具备写-写、写-读排他特性
lock tables table_name write;

# 查看当前库中场景过的表锁，in_use>0表示目前正在使用的表锁
show open tables where in_use > 0;

# 释放已获取到的锁
unlock tables;
~~~

以上是`MyISAM`存储引擎中，获取表级别的共享锁和排他锁的方式，但这里的关键字是`READ、WRITE`，翻译过来就是读、写的意思，但是上面说过共享锁不是读锁，排他锁不是写锁，不要混了

> 但是在`MyISAM`存储引擎中获取了锁还需要自己手动去释放锁，否则会造成死锁的现象出现，因为如果不手动释放锁，就算事务结束也不会自动释放锁，除非当前的数据库连接中断时才会释放

你可以根据上面步骤来做做实操，可以证明出`MyISAM`表锁显式获取后，一定要自己手动释放，否则结合数据库连接池，由于数据库连接是长存的，就会导致表锁一直被占用

> 而且当你加入`read`读锁后，再尝试加入`write`写锁后，会发现无法获取锁，当前线程会陷入阻塞，反过来也一样

对`InnoDB、MyISAM`存储引擎中的表锁做了简单介绍后，除了基本的表锁，还有几种的表锁，如：元数据锁、意向锁、自增锁、全局锁，接下来聊一聊这些特殊的锁

-----

## 元数据锁`Meta Data Lock`

`Meta Data Lock`元数据锁，也被称为`MDL`锁，是基于表的元数据加锁。

我们知道：

- 表锁是基于整张表加锁
- 行锁是基于某张表的一条数据进行加锁
- `MDL`锁就是基于`.frm`文件中的元数据加锁的

**所有的存储引擎的表都会存在一个`.frm`文件，这个文件中主要 存储表的结构`DDL语句`**

这种锁是在`MySQL5.5`版本后才支持的，一般我们也用不上，所以不需要手动去获取锁，这个锁主要是用于：更改表结构时使用，比如你要向一张表创建/删除一个索引、修改一个字段的名称/数据类型、增加/删除一个表字段等这些情况

因为当你在对表结构进行更改时，假设此时有其他的事务来对表进行`crud`操作，自然就会出现问题，比如刚删除了一个表字段，结果另一个事务是按照没有删除字段的表结构来进行插入数据，所以就要加`MDL`锁，加了之后整张表就不允许在更改表结构时，其他事务来做任何的操作

-----

## 意向锁`Intention Lock`

`InnoDB`存储引擎是一种支持多粒度锁的存储引擎，而意向锁就是`InnoDB`中为了支持多粒度的锁，为了兼容行锁和表锁而设计的，看个例子：

> 假设一张表有一千万条数据，现在有一个`事务A`对`id=88888`这条数据加一个行锁，此时再来一个`事务B`，想要获取这张表的表级别写锁，经过前面的讲解，写锁一定是排他锁，在同一时刻中，只允许获取到写锁的一个事务进行写操作，但是如果表中存在其他已经获取到锁的事务，目前获取到写锁的事务就无法满足排他性了

由于`事务A`对`id=88888`的数据加了行锁，那`事务B`获取表锁时，是不是要判断一下表中是否存在其他事务的操作，就是为了保证这个获取到表锁的事务的排他性，但是因为`InnoDB`中有行锁的概念，所以表中任何一行数据上都有可能存在事务加锁操作，所以为了精确知道答案，`MySQL`会将整张表的所有数据都遍历一遍，然后逐条查看是否存在锁，效率很低很低

效率低就算了，而且还会产生一些致命的问题，就是如果`MySQL`读取数据读到某一个位置，加入第`574w`条数据时，发现前面的数据都没有加锁，那么它还会继续往下遍历查看，但是`MySQL`是支持并发事务的，所以当你`MySQL`在往下去扫描数据的时候，是有可能有其他事务过来对已经扫描过的数据进行加锁的，这样很可能导致死循环了，所以行锁和表锁之间就出现了兼容的问题

所以意向锁就出现了，意向锁是一种特殊的表锁，其实他就是一种挂牌的思想，像出租车意义，挂个牌表示现在是空车还是载客的情况

例如：`事务A`对`id=88888`这个数据加行锁前，就会先加入一个表级别的意向锁，如果`事务A`对`id=88888`这个数据加行级的读锁，就会先添加一个表级别的意向共享锁，如果`事务A`要加行级别的写锁，就是同理的

此时`事务B`尝试去获取表级锁时，就先看表上是否有意向锁，如果有再去判断一下是否和自身是否冲突，例如：表上有一个意向共享锁，目前`事务B`要获取的是表级别的读锁，那自然是不会冲突的，是可以获取的，但是如果要获取一个表级的写锁时，是会出现冲突的，`事务B`就会陷入阻塞，直到`事务A`释放锁后，即事务提交后，才能继续往下执行

-----

## 自增锁`AUTO-INC Lock`

自增锁就是专门为了提升自增ID的并发插入性能而设计的，通常情况下我们在建表的时候，都会对一张表的主键设置自增特性，如下：

~~~sql
create table table_name (
    user_id not null auto_increment,
    .....
) engine  = InnnoDB;
~~~

对一个字段设置`auto_increment`自增后，意味着后续插入数据时无序为其赋值，系统会自动赋上按顺序自增的值，但是假如现在表中最大的是`id=80`，如果有两个并发事务一起对表执行插入语句，由于是并发执行的原因，所以可能导致插入两条`id=81`的数据，因此这里必须要加上一个排他锁，确保并发插入时的安全性，但由于锁的原因，插入效率也就因此降低了，毕竟将所有写操作串行化了

> 所以，为了改善插入数据时的性能，自增锁就诞生了，自增锁也是一种特殊的表锁，但是它仅为具备`auto_increment`自增字段的表服务，同时自增锁也分为不同的级别，可以通过`innodb_autoinc_lock_mode`参数控制

- `innodb_autoinc_lock_mode=0`：传统模式
- `innodb_autoinc_lock_mode=1`：连续模式(`MySQL8.0`以前的默认模式)
- `innodb_autoinc_lock_mode=2`：交错模式(`MySQL8.0`以后的默认模式)

当然这三种模式又是什么含义呢？首先我们先明白`MySQL`中可能出现的三种插入类型：

- 普通插入：通过`insert into table_name(...) values(....)`这种方式插入，这也是比较常用的方式
- 批量插入：通过`insert .... select .....`这种方式批量插入查询出的数据
- 混合插入：通过`insert into table_name(id , .....) values (1, ....), (null, ....), (3, ....)`这种方式插入，其中一部分指定`id`，一部分不能指定

**自增锁主要负责维护并发事务下自增列的顺序**，即每个事务想要向表中插入数据时，都要先获取自增锁先分配一个自增的顺序值，但是在不同模式下的自增锁有些许不同

> **传统模式**：`事务A`获取自增锁插入数据，`事务B`也要插入数据，此时`事务B`只能陷入阻塞等待`事务A`执行结束释放锁资源之后才能继续执行，即传统模式下的自增锁只允许一条线程执行，这种形式显然性能较低

> **连续模式**：这个模式主要是因为传统模式性能太低而出现的，在这个模式中，对于能够提确定数量的插入语句，就不会再获取自增锁。
>
> 即对于普通插入类型的语句，因为在插入前就已经确定要插入多少条数据了，因为会直接分配范围自增值
>
> 
>
> 例如：
> `事务A`通过`insert .... into`语句插入十条数据，目前表中存在的最大是`id=88`，那么在连续模式下，`MySQL`会直接将`89~98`这十个自增值分配给`事务A`，因此`事务A`不需要再去获取自增锁，但不获取自增锁就不代表不获取锁，而是改为使用一种轻量级锁`Mutex-Lock`来防止自增值重复分配
>
> 
>
> 对于普通插入类型的操作，由于可以提前确定插入的数据量，因此可以采用“预分配”思想，但如若对于批量插入类型的操作，因为批量插入的数据是基于`SELECT`语句查询出来的，所以在执行之前也无法确定究竟要插入多少条，所以依旧会获取自增锁执行。也包括对于混合插入类型的操作，有一部分指定了自增值，但有一部分需要`MySQL`分配，因此“预分配”的思想也用不上，因此也要获取自增锁执行。

> 交错模式：在交错插入模式中，对于`INSERT、REPLACE、INSERT…SELECT、REPLACE…SELECT、LOAD DATA`等一系列插入语句，都不会再使用表级别的自增锁，而是全都使用`Mutex-Lock`来确保安全性，为什么在这个模式中，批量插入也可以不获取自增锁呢？这跟它的名字有关，目前这个模式叫做交错插入模式，也就是不同事务之间插入数据时，自增列的值是交错插入的，举个例子理解。
>
> 好比事务`T1、T2`都要执行批量插入的操作，因为不确定各自要插入多少条数据，所以之前那种“连续预分配”的思想用不了了，但虽然无法做“连续的预分配”，那能不能交错预分配呢？好比给`T1`分配`{1、3、5、7、9....}`，给`T2`分配`{2、4、6、8、10.....}`，然后两个事务交错插入，这样岂不是做到了自增值即不重复，也能支持并发批量插入？答案是`Yes`，但由于两个事务执行的都是批量插入的操作，因此事先不确定插入行数，所以有可能导致“交错预分配”的顺序值，有可能不会使用，比如`T1`只插入了四条数据，只用了`1、3、5、7`，`T2`插入了五条数据，因此表中的自增值有可能出现空隙，即`{1、2、3、4、5、6、8、10}`，其中`9`就并未使用。



> 虽然我没看过自增锁这块的源码，但交错插入模式底层应该是我推测的这种方式实现的，也就是利用自增列的步长机制实现，不过由于插入可能会出现空隙，因此对后续的主从复制也有一定影响（以后再细聊）。
>
> *不过相对来说影响也不大，虽然无法保证自增值的连续性，但至少能确保递增性，因此对索引的维护不会造成额外开销*。

## 全局锁

全局锁就是一种尤为特殊的表锁，将它称为库锁更合适，因为全局锁是基于整个数据库来加锁的，加上全局锁之后，整个数据库只能允许读，不允许做出任何写操作，一般全局锁是在对整库做数据备份时使用

~~~sql
-- 获取全局锁的命令
FLUSH TABLES WITH READ LOCK;

-- 释放全局锁的命令
UNLOCK TABLES;

~~~

# MySQL行锁

为了尽可能提升数据库的整体性能，在每次加锁的时候，锁的范围肯定是越小越好，举个例子：

> 假设此时有`1000`个请求，要操作`user_table`表中的数据，如果以表粒度来加锁，假设第一个请求获取到的是排他锁，也就意味着其他`999`个请求都是需要阻塞等待的，这效率很低

虽然这`1000`个请求操作`user_table`表，但这些请求中至少有百分之90以上要操作的都是表中不同的行数据，因此每个请求都获取表级锁，显然效率很低，而`InnoDB`存储引擎考虑到了这个问题，所以实现了更细粒度的锁，即行锁`其实就是范围更小的锁`

----

## 表锁和行锁之间的关系

表锁和行锁之间的关系，我们举个生活上的例子来快速理解一下，就例如是住酒店的一个案例，一张表就相对于一间酒店，每个事务/每个请求就相当于一个个的旅客，表锁就相当于是一个旅客入住酒店之后，为了保证旅客住宿的安全，酒店会锁上酒店大门，这个就相当于是表锁，当有了表锁之后，就是酒店大门锁上之后，如果后面还有旅客想入住，就是如果除了当前拥有表锁的事务之外的其他事务，只有等酒店大门开锁后，即表锁释放后，其他旅客才能入住

> 但其实这样设计很不合理，表锁是每次只能住一个旅客，效率极低

行锁就相当于是酒店里每一个房间的房门锁，使用这个门锁就不用每次都锁酒店大门，每次只能入住一个酒店，只要在酒店容量范围内，想来多少个就来多少个，每个住户入住之后只需要把自己的门锁锁上即可，所以行锁的性能更优

-----

## InnoDB的行锁实现

在`MySQL`诸多存储引擎中，仅有`InnoDB`存储引擎支持行锁，因为`InnoDB`存储引擎支持聚簇索引，`InnoDB`如果能命中索引数据，就会加行锁，无法命中就会加表锁

> 在`MySQL索引原理篇`中讲过，`InnoDB`会将表数据存储在聚簇索引中，每条行数据都会存储在树中的叶子节点上，因此行数据是分开的，所以可以对每一条数据上锁，但其他存储引擎大部分都不支持聚簇索引，表数据都是一起存储在一块的，所以只能基于整个表数据上锁，这就是为什么其他存储引擎不支持行锁的原因

## 记录锁`Record Lock`

实际上就是行锁，一行表数据、一条表记录本身就是一个意思，一条记录就是一行数据，所以行锁也叫记录锁

如何使用行锁呢？

~~~sql
# 获取行级别的 共享锁
select * from user_table where user_id = 1 lock in share mode;

# 获取行级别 排他锁
select * from user_tbale where user_id = 1 for update;
~~~

想要使用`InnoDB`的行锁就是这样写的，如果你的`SQL`可以命中索引数据，那自然加的就是行锁，反之就是表锁

## 间隙锁`Gap Lock`

间隙锁是对行锁的一种补充，主要是用来解决幻读问题，首先理解一下什么是间隙：

~~~SQL
SELECT * FROM `zz_users`;
+---------+-----------+----------+----------+---------------------+
| user_id | user_name | user_sex | password | register_time       |
+---------+-----------+----------+----------+---------------------+
|       1 | 熊猫      | 女       | 6666     | 2022-08-14 15:22:01 |
|       2 | 竹子      | 男       | 1234     | 2022-09-14 16:17:44 |
|       3 | 子竹      | 男       | 4321     | 2022-09-16 07:42:21 |
|       4 | 猫熊      | 女       | 8888     | 2022-09-27 17:22:59 |
|       9 | 黑竹      | 男       | 9999     | 2022-09-28 22:31:44 |
+---------+-----------+----------+----------+---------------------+

~~~

上述表中的最后两条数据，`id`字段从`4`跳到了`9`，那么`4~9`两者之间的范围就称为间隙，而间隙锁则主要锁定的是这块范围

> 那为什么说间隙锁是用来解决幻读问题的呢？
>
> 因为幻读的概念是：一个事务在执行中，然后另一个事务插入了一条数据，从而导致第一个事务操作完成之后发现结果与预想的不一致，跟产生了幻觉一样
>
>  
>
> 举个例子：
>
> 将`id>3`的用户密码重置为`1234`，因此`事务A`先查询到`id>3`的`4、9`这两条数据并上锁，然后开始更改用户密码，但是此时来了个`事务B`，`事务B`过来插入了一条`id=6、password=7777`的数据且提交了，等`事务A`修改完`4、9`这两条数据后，再去查询`id>3`的这数据时，会发现`id=6`的这条数据没有被重置密码
>
>  
>
> 因为`事务B`新增数据并提交了事务，`事务A`在操作的时候没有这条数据，等操作完之后再回去查询数据就会发现多了一条`id=6`的数据，且没有被重置，就跟产生了幻觉有一样，对于这种新增的数据，专业的叫法是幻影数据

> 所以为了防止出现安全问题，`事务A`在操作前会对目标数据加锁，但在`事务A`执行时，这条幻影数据还不存在，因此就会出现一个新问题，就是不知道把锁加在哪里，毕竟想要对`id=6`这条数据枷锁，就是加了个寂寞

锁是肯定要加的，但是怎么加呢？

加普通行锁是无法解决这个问题的，而且也不能加表锁，效率太低，所以间隙锁就出来了，主要就是对间隙区域加锁

举个例子：

~~~sql
select * from zz_users where user_id = 6 lock in share mode;
~~~

> ![间隙锁实验](https://p1-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/4719390a89e249f59ec6a72d64ad14f5~tplv-k3u1fbpfcp-jj-mark:3024:0:0:0:q75.awebp#?w=1698&h=755&s=104827&e=png&b=0c0c0c)
>
> <u>图片来源于网络</u>

这条加锁的`SQL`看起来似乎不太合理，因为`id=6`的数据还没有在表中，这个新增数据的事务都还没有执行。

这个加的是间隙锁，会锁定`4~9`这个区间，但是不包含`4和9`这两个端点位置，因为间隙锁遵循左右开区间的原则，看看上面的图片案例即可

**当对一个不存在的数据加锁后，默认就是锁定前后两条数据之间的区间，当其他事务再尝试向该区域插入数据时，就会陷入阻塞，只有当持有间隙锁的事务结束后，才能继续执行插入操作**

间隙锁加在不同的位置锁定的范围也不一样，如果加在两条数据之间，那么锁定的区域就是两条数据之间的间隙，如果加在上表`id=1`的数据上，锁定的区域是`1~1`，即只锁定`id=1`的这一行数据，毕竟间隙锁的作用是为了保护可能插入的行，而不是已有的行，因此在这里不会锁定其他不存在的行

> `InnoDB`默认的行锁算法为临建锁，所以实际情况下，对`id=1`的数据加锁，锁定的区域是`无穷小到1之间的区域`，如果加在`id=9`之后，锁定的区域就是`9之后到无穷大的区域`

## 临建锁`Next-Key Lock`

临建锁是间隙锁的`plus`版本，或者是记录锁+间隙锁的锁

- 记录锁：锁定的范围是表中具体的一条行数据
- 间隙锁：锁定的范围是左右开区间，但不包含当前这一条真实数据，只锁间隙区域

临建锁是两者的结合体，加锁后，即锁定左开右闭的区间，也会锁定当前行数据，还是以上述标为例：

> ![临键锁实验](https://p9-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/6a99c6769d064bf88d13a8c6d146122a~tplv-k3u1fbpfcp-jj-mark:3024:0:0:0:q75.awebp#?w=1690&h=680&s=93227&e=png&b=0c0c0c)

基于`id=9`这条数据加锁，除开锁定了`4~9`这个区间外，对于`id=9`这条数据也锁定了，因为在`事务B`中尝试对`id=9`的数据修改时，会让事务陷入阻塞

> 临键锁的注意点：当原本持有锁的`事务A`结束后，`事务B`会执行插入操作，这时锁会被`事务B`获取，当你再尝试开启一个新的事务`事务C`，再次获取相同的临键锁时，是无法获取的，只能等`事务B`结束后才能获取（因为临建锁包含了记录锁，虽然间隙锁可以同时由多个事务持有，但排他类型的记录锁只允许一个事务持有）。



> 实际上在`InnoDB`中，除开一些特殊情况外，当尝试对一条数据加锁时，默认加的是临键锁，而并非记录锁、间隙锁。



> 也就是说，在前面举例幻读问题中，当`事务A`要对`ID>3`的用户做密码重置，锁定`4、9`这两条行数据时，默认会加的是临键锁，也就是当事务`事务B`尝试插入`ID=6`的数据时，因为有临建锁存在，因此无法再插入这条“幻影数据”，也就至少保障了`事务A`事务执行过程中，不会碰到幻读问题。

## 插入意向锁`Insert Intention Lock`

是一种间隙锁，和意向锁没有什么关系，这种锁是一种隐式锁，也就是我们无法手动去获取的锁，通常在`MySQL`中插入数据时，是并不会产生锁的，但在插入前会先简单判断一下，当前事务要插入的位置有没有存在间隙锁或临建锁，如果存在，当前插入数据的事务需要阻塞等待，直到拥有临建锁的事务提交

> 当事务执行插入语句阻塞时，就会生成一个插入意向锁，表示当前事务想对一个区间插入数据（目前的事务处于等待插入意向锁的状态）。

当持有原本持有临建锁的事务提交后，当前事务即可以获取插入意向锁，然后执行插入操作，当此时如若又来一个新的事务，也要在该区间中插入数据，那新的事务会阻塞吗？答案是不会，可以直接执行插入操作，`Why`？

> 因为在之前的`SQL执行篇-写入SQL执行流程`中曾说到过，对于写入`SQL`都会做一次唯一性检测，如果要插入的数据，与表中已有的数据，存在唯一性冲突时会直接抛出异常并返回。这也就意味着：如果没抛出异常，就代表着当前要插入的数据与表中数据不存在唯一性冲突，或表中压根不存在唯一性字段，可以允许插入重复的数据。

简单来说就是：能够真正执行的插入语句，绝对是通过了唯一检测的，因此插入时可以让多事务并发执行，同时如果设置了自增`ID`，也会获取自增锁确保安全性，所以当多个事务要向一个区间插入数据时，插入意向锁是不会排斥其他事务的，从这种角度而言，插入意向锁也是一种共享锁

## 行锁的粒度粗化

有一点要值得注意：行锁并不是一成不变的，行锁会在某些特殊情况下发生粗化，主要有两种情况：

- 在内存中专门分配了一块空间存储锁对象，当该区域满了后，就会将行锁粗化为表锁。
- 当做范围性写操作时，由于要加的行锁较多，此时行锁开销会较大，也会粗化成表锁。

当然，这两种情况其实很少见，因此只需要知道有锁粗化这回事即可，这种锁粗化的现象其实在`SQLServer`数据库中更常见，因为`SQLServer`中的锁机制是基于行记录实现的，而`MySQL`中的锁机制则是基于事务实现的（后续《事务与锁原理篇》详细剖析）

# 页面锁、乐观锁、悲观锁

## 页面锁

页面锁`Berkeley DB`存储引擎支持的一种锁粒度，由于`BDB`存储引擎被`Oracle`收购的原因，`MySQL5.1`以后不再直接性地支持该存储引擎，需要自己整合，因此页锁见的也比较少，了解即可

- 表锁：以表为粒度，锁住的是整个表数据
- 行锁：以行为粒度，锁住的是一条数据
- 页锁：以页为粒度，锁住的是一页数据

----

## 乐观锁

乐观锁是无锁思想

- 乐观锁：每次执行都认为只会有自身一条线程操作，因此无需拿锁直接执行
- 悲观锁：每次执行都认为会有其他线程一起来操作，因此每次都需要先拿锁再执行

这种无锁技术或者说乐观锁机制，一般都是基于`CAS`思想实现的，而在`MySQL`中则可以通过`version`版本号+`CAS`的形式实现乐观锁，也就是在表中多设计一个`version`字段，然后在`SQL`修改时如下形式操作：

~~~sql
update ... set version = version + 1 .... where ... and version = version;
~~~

也就是每条修改的`SQL`都在修改之后对`version`字段加1，例如`事务A、事务B`这两个事务并发执行时，当`事务B`执行成功并提交后，就会对`version+1`，因此`事务A`的`version=version`这个条件就不成立了，最终会放弃执行，因为`version`已经被其他事务修改了

**乐观锁适用于读大于写的业务场景，频繁写库的业务则不适合加乐观锁**

----

## 悲观锁

在数据库中想要使用悲观锁，就对一个事务加排他锁`for update`即可

悲观锁就是每次执行时都会加锁再执行

由于临建锁也会锁定相应的行数据，因此上表中也不再重复赘述记录锁，临建锁兼容的 记录锁都兼容，同理，冲突的记录锁也会冲突，再来看看标记别的锁对比：

| 表级锁对比 | 共享意向锁 | 排他意向锁 | 元数据锁 | 自增锁 | 全局锁 |
| ---------- | ---------- | ---------- | -------- | ------ | ------ |
| 共享意向锁 | 兼容       | 兼容       | 冲突     | 兼容   | 冲突   |
| 排他意向锁 | 兼容       | 兼容       | 冲突     | 兼容   | 冲突   |
| 元数据锁   | 冲突       | 冲突       | 冲突     | 冲突   | 冲突   |
| 自增锁     | 冲突       | 冲突       | 冲突     | 冲突   | 冲突   |
| 全局锁     | 兼容       | 冲突       | 冲突     | 冲突   | 冲突   |

放眼望下来，其实会发现表级别的锁，会有很多很多冲突，因为锁的粒度比较大，因此很多时候都会出现冲突，但对于表级锁，咱们只需要关注共享意向锁和共享排他锁即可，其他的大多数为`MySQL`的隐式锁（在这里，共享意向锁和排他意向锁，也可以理解为`MyISAM`中的表读锁和表写锁）。

> 最后再简单的说一下，表中的冲突和兼容究竟是啥意思？冲突的意思是当一个事务`T1`持有某个锁时，另一个事务`T2`来请求相同的锁，`T2`会由于锁排斥会陷入阻塞等待状态。反之同理，兼容的意思是指允许多个事务一同获取同一个锁。

### MySQL5.7版本中新增的共享排他锁

对于这条是最后补齐的，之前漏写了这种锁类型，在`MySQL5.7`之前的版本中，数据库中仅存在两种类型的锁，即共享锁与排他锁，但是在`MySQL5.7.2`版本中引入了一种新的锁，被称之为`(SX)`共享排他锁，这种锁是共享锁与排他锁的杂交类型，至于为何引入这种锁呢？聊它之前需要先理解`SMO`问题：

> 在`SQL`执行期间一旦更新操作触发`B+Tree`叶子节点分裂，那么就会对整棵`B+Tree`加排它锁，这不但阻塞了后续这张表上的所有的更新操作，同时也阻止了所有试图在`B+Tree`上的读操作，也就是会导致所有的读写操作都被阻塞，其影响巨大。因此，这种大粒度的排它锁成为了`InnoDB`支持高并发访问的主要瓶颈，而这也是`MySQL 5.7`版本中引入`SX`锁要解决的问题。

那想一下该如何解决这个问题呢？最简单的方式就是减小`SMO`问题发生时，锁定的`B+Tree`粒度，当发生`SMO`问题时，就只锁定`B+Tree`的某个分支，而并不是锁定整颗`B+`树，从而做到不影响其他分支上的读写操作。

> 那`MySQL5.7`中引入共享排他锁后，究竟是如何实现的这点呢？首先要弄清楚`SX`锁的特性，它不会阻塞`S`锁，但是会阻塞`X、SX`锁，下面展开来聊一聊。

在聊之前首先得搞清楚`SQL`执行时的几个概念：

- 读取操作：基于`B+Tree`去读取某条或多条行记录。
- 乐观写入：不会改变`B+Tree`的索引键，仅会更改索引值，比如主键索引树中不修改主键字段，只修改其他字段的数据，不会引起节点分裂。
- 悲观写入：会改变`B+Tree`的结构，也就是会造成节点分裂，比如无序插入、修改索引键的字段值。

在`MySQL5.6`版本中，一旦有操作导致了树结构发生变化，就会对整棵树加上排他锁，阻塞所有的读写操作，而`MySQL5.7`版本中，为了解决该问题，对于不同的`SQL`执行，流程就做了调整。

#### MySQL5.7中读操作的执行流程

- ①读取数据之前首先会对`B+Tree`加一个共享锁。
- ②在基于树检索数据的过程中，对于所有走过的叶节点会加一个共享锁。
- ③找到需要读取的目标叶子节点后，先加一个共享锁，释放步骤②上加的所有共享锁。
- ④读取最终的目标叶子节点中的数据，读取完成后释放对应叶子节点上的共享锁。

#### MySQL5.7中乐观写入的执行流程

- ①乐观写入之前首先会对`B+Tree`加一个共享锁。
- ②在基于树检索修改位置的过程中，对于所有走过的叶节点会加一个共享锁。
- ③找到需要写入数据的目标叶子节点后，先加一个排他锁，释放步骤②上加的所有共享锁。
- ④修改目标叶子节点中的数据后，释放对应叶子节点上的排他锁。

#### MySQL5.7中悲观写入的执行流程

- ①悲观更新之前首先会对`B+Tree`加一个共享排他锁。
- ②由于①上已经加了`SX`锁，因此当前事务执行过程中会阻塞其他尝试更改树结构的事务。
- ③遍历查找需要写入数据的目标叶子节点，找到后对其分支加上排他锁，释放①中加的`SX`锁。
- ④执行`SMO`操作，也就是执行悲观写入操作，完成后释放步骤③中在分支上加的排他锁。

如果需要修改多个数据时，会在遍历查找的过程中，记录下所有要修改的目标节点。

#### MySQL5.7中并发事务冲突分析

观察上述讲到的三种执行情况，对于读操作、乐观写入操作而言，并不会加`SX`锁，共享排他锁仅针对于悲观写入操作会加，由于读操作、乐观写入执行前对整颗树加的是`S`锁，因此悲观写入时加的`SX`锁并不会阻塞乐观写入和读操作，但当另一个事务尝试执行`SMO`操作变更树结构时，也需要先对树加上一个`SX`锁，这时两个悲观写入的并发事务就会出现冲突，新来的事务会被阻塞。

> 但是要注意：当第一个事务寻找到要修改的节点后，会对其分支加上`X`锁，紧接着会释放`B+Tree`上的`SX`锁，这时另外一个执行`SMO`操作的事务就能获取`SX`锁啦！

其实从上述中可能得知一点：`MySQL5.7`版本引入共享排他锁之后，解决了`5.6`版本发生`SMO`操作时阻塞一切读写操作的问题，这样能够在一定程度上提升了`InnoDB`表的并发性能。

> 最后要注意：虽然一个执行悲观写入的事务，找到了要更新/插入数据的节点后会释放`SX`锁，但是会对其上级的叶节点（叶分支）加上排他锁，因此正在发生`SMO`操作的叶分支，依旧是会阻塞所有的读写行为！

上面这句话啥意思呢？也就是当一个要读取的数据，位于正在执行`SMO`操作的叶分支中时，依旧会被阻塞。
